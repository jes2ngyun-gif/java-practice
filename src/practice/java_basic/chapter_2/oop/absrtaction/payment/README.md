### 문제 설명
쇼핑몰에는 여러 결제 방식이 있다.
- 카드 결제
- 계좌 이체

둘 다
- 결제 금액
- 결제 정보 출력

이라는 공통점은 있지만,

실제 결제 방식은 다르다.

따라서

> 결제는 반드시 해야 한다.

라는 규칙만 만들고,

구체적인 구현은 각 결제 방식에게 맡겨본다.

### 핵심 원리
상속에서의 부모는
> 공통 기능을 물려주는 존재

였다.

하지만 추상화에서의 부모는 한 단계 더 발전해서
> **반드시 지켜야 할 규칙까지 정의하는 존재**

가 된다.

---
### 중요한 것들!!
#### 1. abstract가 왜 필요한가??
만약에 부모가
```
public void pay() {
}
```
라고 비워두면,

자식이 구현하지 않아도 프로그램이 돌아간다.

그러면

```
카드 결제인데 실제 결제 기능이 없음
```
같은 이상한 객체가 만들어질 수 있다.

그래서

```
public abstract void pay();
```
라고 선언하면
> 결제 클래스라면 반드시 pay()를 구현해!!

라고 강제로 규칙을 만들 수 있다.

---
#### 2. Payment는 객체를 만들 수 없다.
```
Payment payment = new Payment(10000);
```
NONONONO 절대 불가능

왜냐하면 Payment는 **설계도 역할만 하기 때문**이다.

실제 결제 방식은

```
CardPayment
BankTransferPayment
```
같은 **자식이 담당한다.**
